This might be something wrong with our defaults for how much we wait
between one ``read()`` and the following ‒ e.g. see
https://bitbucket.org/steko/totalopenstation/src/27a645a3158d/scripts/totalopenstation-gui.py#cl-706

You might try changing the ``sleep()`` time to a smaller amount of
seconds ‒ if you have no idea of how to do it we can explain in more
detail.

> What is it really that the 'process data option' does? Is it simple to
> implement a new model for the 'process data' option..? I'd like to

We would be happy if you could contribute to the development of TOPS
with a new input format! The basic functionality of "process data" is
very simple: it reads the original data and performs some filters until
we get a standard representation of each data point.

You can take a look at
https://bitbucket.org/steko/totalopenstation/src/27a645a3158d/totalopenstation/formats/leica_tcr_705.py
and the respective sample data in
https://bitbucket.org/steko/totalopenstation/src/27a645a3158d/sample_data/leica_tcr_705 to have an idea.

In fact, I have a bunch of GSI files on my hard disk, but I've never had
the time to look at them: it should not be difficult to write an "input
parser" for it.
